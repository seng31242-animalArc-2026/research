# Contributing

## Branching Strategy

This repository uses a simple placeholder branching strategy until the team defines a formal workflow.

- `main` should contain stable, reviewed work.
- Create feature branches from `main` for new work.
- Use short, descriptive branch names, such as `feature/add-project-docs`, `fix/update-readme`, or `docs/contributing-guide`.
- Open a pull request before merging changes into `main`.
- Keep branches focused on a single change or related set of changes.

## Commit Convention

Use clear, concise commit messages that describe the change.

Suggested format:

```text
type: short summary
```

Common commit types:

- `feat` - Adds a new feature or capability.
- `fix` - Fixes a bug or incorrect behavior.
- `docs` - Updates documentation.
- `chore` - Handles maintenance or project setup.
- `test` - Adds or updates tests.
- `refactor` - Changes code structure without changing behavior.

Examples:

```text
docs: add initial repository README
chore: add baseline gitignore
feat: add data import script
```

Update this file when the team agrees on a more specific branching model, review process, or commit standard.
